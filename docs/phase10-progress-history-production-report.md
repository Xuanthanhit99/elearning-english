# Phase 10 - Progress, History and Unified Activity Timeline

## Production Decision

Status: READY_WITH_LIMITATIONS

Phase 10 is implemented by reusing the existing Progress module and domain data sources. No duplicate Progress or History module was created.

## Existing Code Reused

- Backend ProgressModule, ProgressController, ProgressService.
- Existing DashboardService for summary, XP, streak, recommendation and learning path data.
- Existing SettingsQueryService and shared user timezone utility from Phase 9.
- Existing domain session/progress tables:
  - UserWordProgress
  - GrammarLessonProgress
  - ReadingSession
  - ListeningSession
  - SpeakingSession
  - WritingSession
  - LessonProgress
  - XpTransaction

## Backend Completed

- Added unified progress query DTOs and filters.
- Added authenticated progress overview API:
  - GET /progress
- Added skill progress APIs:
  - GET /progress/skills
  - GET /progress/skills/:skill
- Added in-progress API:
  - GET /progress/in-progress
- Added unified history API:
  - GET /progress/history
- Added activity detail API:
  - GET /progress/activities/:activityId
- Preserved existing lesson/course progress APIs.
- All new read APIs use authenticated user from request context.
- No frontend userId is required or accepted.
- Read APIs do not create XP, progress, streak, mission or notification side effects.

## Unified Timeline Contract

The timeline is built by merging domain records instead of using XP ledger alone.

Each activity contains:

- activityKey
- type
- skill
- title
- description
- status
- entityType
- entityId
- sessionId
- score
- accuracy
- xpEarned
- durationSeconds
- startedAt
- completedAt
- occurredAt
- action
- metadata

## Filters and Pagination

Supported on GET /progress/history:

- skill
- status
- activityType
- range
- from
- to
- cursor
- limit

Cursor pagination is stable by occurredAt and activityKey.

## Timezone

- User timezone is read from settings.
- Date range filtering uses the shared timezone utility.
- History groups are grouped by user-local date.

## Known Limitations

- Activity detail currently supports the activity keys generated by the unified timeline sources.
- Progress skill route accepts the existing LearningSkill enum values from Prisma.
- The unified timeline is query-merged at read time. This avoids migration/backfill risk for Phase 10, but a dedicated event table can be considered later if activity volume becomes very large.

## Migration Decision

No database migration was required for Phase 10.

## Backfill Decision

No backfill was required because the timeline reads from existing source-of-truth domain tables.

## Cache Decision

No cache was added in Phase 10. The current implementation uses bounded range queries and cursor pagination.
